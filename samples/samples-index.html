dart-squid Sample #1

NOTE: As of recent Dartium builds (based on Chrome v25.x), the refresh of the SVG objects (repaint regions) is completely messed up.  For some reason, the browser is no longer repainting anything outside of a rectangular area whose extent is defined having a width ending at the right-most "menu button" widget and a height that has no relation to anything.  But, the browser WILL repaint SVG items anywhere on the screen IF you manually resize the browser window.  It is bugs like this that prevent any real progress on this component set.  It has been one issue after another and constantly-changing browser and/or Dart-library issues breaking what used to work just fine in a native JS-only implementation of this Widget set.  As of December, 2012, if some stability does not come to the browser and Dart SVG code and rendering, this project is most likely going to be abandon in favor of native JS code, a different target-browser, a native Windows WPF approach, or some such alternative.

Sample #1 (SVG wrapped in HTML)

This is a very basic visual demonstration of the dart-squid Widgets in action. Play around a bit and you will notice some alignment features (between widgets and related to browser-window boundaries), some position-constraint features that invoke during movement, the ability to hide/show Widgets (including any hierarchically contained), multi-select, re-size/movement, and so forth.

Sample #1 (SVG)

Note: Although the reported Dart Issue 2977 (ElementImplementation behavior fluctuates based on containing doc type (HTML vs. SVG) is marked as "fixed", there is still some implementation difference between HTML-"wrapped" SVG and standalone.  The standalone SVG STILL does not work (hits an error stating: "Exception: NoSuchMethodError : method not found: '$dom_children'").   Sadly, this type of continual problems in the Dart SVG implementations are making the goals of this project impossible to implement.

Another issue with the non-wrapped SVG version is odd behavior with event-code removal, thus making the selection (and de-selection) of widgets unpredictable or impossible. Again, this may be very much related to issue 2977 (with regards to the element hierarchy in DART) since, by simply wrapping this SVG doc with a dummy-HTML file, the behavior within the embedded SVG is totally as expected and predictable. Clearly something is screwy in the Dart HTML library with regards to SVG elements that are "standalone" vs. "wrapped".

The ultimate goal is to remove the need for an outer HTML "wrapper" and have the entire "application" be just an SVG document with all embedded functionality and interactivity contained in that standalone SVG document. This does work (these widgets were first implemented in JS prior to Dart's existence and it worked fine there), though Dart's API/Library issues prevent this currently.

dart-squid SVG-Widget-Application Sample #1

Sample #1 (SVG wrapped in HTML)

This is a very basic visual demonstration of the dart-squid Widgets in action. Play around a bit and you will notice some alignment features (between widgets and related to browser-window boundaries), some position-constraint features that invoke during movement, the ability to hide/show Widgets (including any hierarchically contained), multi-select, re-size/movement, and so forth.

NOTE: As of recent Dartium builds (based on Chrome v28.x+), visual painting / refresh (repaint regions) of those initially-showing SVG objects whose on-screen position is dependent on browser-window-metrics will not occur until you trigger a "resize" operation manually by either resizing the browser-window up toggling page-zoom-factor.

WORKAROUNDS — JUN-2013 UPDATE: here are latest "workarounds" to the initial-SVG-Repaint-Region problem:

  • DARTIUM as of latest build (Version 29.0.1530.0 (204370)) — simply open the Developer-Tools Window prior to refreshing page (Dev-Tools region can be sized as small as you choose) &mdsahs; so long as this is done, all Widgets will repaint / show on initial page display.
  • Chromium / Browsers in general, prior to initial-widget-interaction, simply resize your browser window a bit or cycle the page-zoom (CTRL+, CTRL- keys) and the Widget(s) should all repaint and show.


Sample #1 (as standalone SVG document)

Note: Dart Issue 2977 — originally reported in May, 2012 — (ElementImplementation behavior fluctuates based on containing doc type (HTML vs. SVG) is marked as "fixed", but implementation differences between HTML-"wrapped" SVG and standalone SVG persist.  Standalone SVG document capabilities REMAIN broken and will terminate on error(s) in the Dart library internals.. These Widgets used to work just fine in standalone SVG documents, before the Dart team apparently decided that SVG was a second-class document type and broke this capability.  

The Dart team has since created a NEW issue with no timeline/milestone target for a fix: see Dart Issue 6947 — Support top-level SVG documents. Sadly, these continual problems in the Dart SVG implementations are making the goals of this project impossible to implement.

Another issue that existed in the non-HTML-wrapped-SVG version involved problematic event-handler removal, thus making the selection (and de-selection) of widgets unpredictable or impossible. This may be directly related to issue 6947 (with regards to the element hierarchy in DART) since, by simply wrapping this SVG doc with a dummy-HTML file, the behavior within the embedded SVG is totally as expected and predictable. Clearly something is screwy in the Dart HTML library with regards to SVG elements that are "standalone" vs. "wrapped".

The ultimate goal is to remove the need for an outer HTML "wrapper" and have the entire "application" be just an SVG document with all embedded functionality and interactivity contained in that standalone SVG document. This does work (these widgets were first implemented in JS prior to Dart's existence and it worked fine there), though Dart's API/Library issues prevent this currently.
